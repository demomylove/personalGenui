代理生态系统中的 A2UI (A2UI in the Agent Ecosystem)

Agentic UI 领域发展迅速，优秀的工具不断涌现以解决技术栈的不同部分。A2UI 并不是这些框架的替代品——它是一个专门的协议，用于解决 可互操作、跨平台、生成式或基于模板的 UI 响应 的特定问题。

概览 (At a glance)

A2UI 的方法是将 JSON 作为消息发送给客户端，然后客户端使用渲染器将其转换为原生 UI 组件。LLM 可以即时生成组件布局，或者您可以使用模板。

这使得它像数据一样安全，像代码一样具有表现力。

本页的其余部分将帮助您了解 A2UI 与其他选项的关系。

导航 Agentic UI 生态系统

1. 构建 "Host" 应用程序 UI

如果您正在构建全栈应用程序（用户与之交互的“宿主”UI），除了构建实际的 UI 之外，您还可以利用框架 (AG UI / CopilotKit, Vercel AI SDK, 已经在使用 A2UI 的 Flutter GenUI SDK) 来处理“管道”：状态同步、聊天历史和输入处理。

A2UI 适用的位置： A2UI 是互补的。如果您使用 AG UI 连接宿主应用程序，它可以将 A2UI 用作数据格式，以渲染来自宿主代理以及第三方或远程代理的响应。这为您提供了两全其美的方案：一个丰富的、有状态的宿主应用程序，它可以安全地渲染来自它不控制的外部代理的内容。

  • A2UI with A2A: 您可以通过 A2A 直接发送到客户端前端。
  • A2UI with AG UI: 您可以通过 AG UI 直接发送到客户端前端。
  • A2UI with REST, SSE, WebSockets 和其他传输也是可行的，但尚未可用。

2. UI 作为一种 "资源" (MCP Apps)

Model Context Protocol (MCP) 最近引入了 MCP Apps，这是一个整合了 MCP-UI 和 OpenAI 优秀工作的新标准，使服务器能够提供交云接口。这种方法将 UI 视为工具可以返回的一种资源（通过 ui:// URI 访问），通常在沙盒 iframe 中渲染预构建的 HTML 内容，以确保隔离和安全。

A2UI 有何不同： A2UI 采用“原生优先 (native-first)”的方法，这与 MCP Apps 的资源获取模型不同。A2UI 代理不检索要在沙盒中显示的不透明负载，而是发送原生组件的蓝图。这使得 UI 可以完美继承宿主应用程序的样式和无障碍功能。在多代理系统中，编排代理可以轻松理解来自子代理的轻量级 A2UI 消息内容，从而实现代理之间更流畅的协作。

3. 特定平台的生态系统 (OpenAI ChatKit)

像 ChatKit 这样的工具为了在 OpenAI 生态系统中专门部署代理提供了高度集成、优化的体验。

A2UI 有何不同： A2UI 专为在 Web、Flutter 和原生移动设备上构建自己的代理界面的开发人员设计，或者为需要在信任边界之间进行通信的企业网格（如 A2A）设计。A2UI 赋予客户端更多控制样式的权力，而牺牲了代理的控制权，以便与宿主客户端应用程序实现更大的视觉一致性。
